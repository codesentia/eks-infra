## Context

The `thor` cluster is failing to initialize pods because the AWS VPC CNI plugin is not running. EKS clusters require the VPC CNI add-on to assign pod IPs from VPC subnets. Without it, the kubelet reports `NetworkPluginNotReady` and pods remain in `Pending` or `ContainerCreating` state indefinitely.

The VPC CNI plugin (`aws-node` DaemonSet in `kube-system` namespace) requires IAM permissions to:
- Describe and attach ENIs to EC2 instances
- Assign secondary IP addresses to ENIs
- Tag ENIs for lifecycle management

Current state: The `iam/node-role.yaml` includes `AmazonEKS_CNI_Policy` as a node-level permission. However, the ClusterConfig does not declare the `vpc-cni` add-on, so eksctl does not install it during cluster creation. The interim approach documented in the architecture was to attach CNI permissions to the node role temporarily, but this was never intended to replace the add-on declaration.

The cluster OIDC provider must exist before the VPC CNI IRSA role can be created (IRSA trust policy references the OIDC issuer URL). However, eksctl requires the IRSA role ARN to be specified in the add-on configuration at cluster creation time. This creates a chicken-and-egg dependency.

Resolution: The OIDC provider association must happen **before** cluster creation, not after. The current Makefile post-create step is too late. eksctl can associate the OIDC provider as part of cluster creation if instructed to do so in the ClusterConfig.

## Goals / Non-Goals

**Goals:**
- Add `addons` section to `clusters/thor.yaml.tpl`: declare `vpc-cni` add-on with version, service account configuration, and IRSA role ARN
- Add `iam.withOIDC: true` to ClusterConfig metadata — eksctl creates OIDC provider during cluster creation
- New CFN template `iam/vpc-cni-role.yaml`: IRSA role for `kube-system:aws-node` service account, trust policy dynamically references OIDC issuer (passed as parameter)
- Update Makefile `post-create-thor` target: after OIDC association, deploy `iam-vpc-cni-role-dev` stack with OIDC issuer as parameter
- Update Makefile `resolve-cluster-thor`: fetch VPC CNI role ARN from CFN exports, substitute into ClusterConfig template
- Remove `AmazonEKS_CNI_Policy` from `iam/node-role.yaml` (node role no longer needs CNI permissions)
- Update cluster-bootstrap runbook with revised order

**Non-Goals:**
- Installing kube-proxy or CoreDNS add-ons — separate change
- Custom VPC CNI configuration (prefix delegation, security groups for pods, custom SNAT) — default configuration is sufficient
- Migrating existing clusters — this change applies to new cluster creation; existing clusters require manual add-on installation

## Decisions

### D1 — Enable OIDC provider creation at cluster creation time

**Decision:** Add `iam.withOIDC: true` to the ClusterConfig metadata block:

```yaml
metadata:
  name: thor
  region: us-east-1
  version: "1.35"

iam:
  withOIDC: true
```

**Rationale:** eksctl automatically creates the IAM OIDC identity provider as part of `eksctl create cluster` when `withOIDC: true` is set. This allows IRSA roles to be created and referenced in add-on configurations at cluster creation time, eliminating the chicken-and-egg dependency. The `post-create-thor` Makefile target becomes redundant for OIDC association (eksctl handles it) but is still needed to store the OIDC issuer URL in Parameter Store for future IRSA role stacks.

---

### D2 — Deploy VPC CNI IRSA role before cluster creation

**Decision:** Revised deployment order:

1. Deploy `vpc-dev`
2. Deploy `iam-node-role-dev`
3. Run temporary cluster creation to get OIDC issuer (or use eksctl utils to pre-create OIDC provider)
4. Deploy `iam-vpc-cni-role-dev` with OIDC issuer as parameter
5. Run `create-cluster-thor` with VPC CNI add-on referencing the IRSA role

**Revised approach:** Use eksctl's `withOIDC: true` to create OIDC provider during cluster creation. The VPC CNI role must be deployed **before** cluster creation, which means we need the OIDC issuer URL before the cluster exists.

**Simpler resolution:** Use a **predictable OIDC issuer URL pattern**. EKS OIDC issuer URLs follow the format:

```
https://oidc.eks.<region>.amazonaws.com/id/<OIDC_ID>
```

However, the `<OIDC_ID>` is randomly generated by AWS at cluster creation time and cannot be predicted.

**Final decision:** Split cluster creation into two phases:

**Phase 1** (this change):
- Create cluster with `withOIDC: true` but **without** the VPC CNI add-on declaration
- eksctl creates OIDC provider during cluster creation
- Post-create: fetch OIDC issuer URL, deploy `iam-vpc-cni-role-dev`, install VPC CNI add-on via `eksctl create addon`

**Phase 2** (future optimization):
- Move VPC CNI add-on declaration into ClusterConfig once IRSA role exists

**Rationale:** This two-phase approach avoids blocking cluster creation on the IRSA role while still achieving IRSA-based CNI permissions. The VPC CNI add-on can be installed immediately after OIDC and IRSA role creation, minimizing downtime.

---

### D3 — VPC CNI version pinning

**Decision:** Specify VPC CNI add-on version explicitly in the `eksctl create addon` command:

```bash
eksctl create addon \
  --cluster thor \
  --name vpc-cni \
  --version v1.19.0-eksbuild.1 \
  --service-account-role-arn <VPC_CNI_ROLE_ARN> \
  --force
```

Use the latest patch version compatible with Kubernetes 1.35 (check AWS EKS add-on compatibility matrix at creation time).

**Rationale:** Version pinning prevents automatic upgrades that could introduce breaking changes. The `--force` flag allows installation even if a default VPC CNI is present (replaces it with the IRSA-enabled version).

---

### D4 — Keep AmazonEKS_CNI_Policy on node role initially; removal is optional

**Decision:** **KEEP** `arn:aws:iam::aws:policy/AmazonEKS_CNI_Policy` in `iam/node-role.yaml` ManagedPolicyArns list during cluster creation. Removal is optional and only after confirming IRSA-based VPC CNI works.

**Rationale:** During cluster creation, EKS installs a default VPC CNI DaemonSet that uses the node IAM role for permissions (not IRSA). If the node role lacks `AmazonEKS_CNI_Policy`, the CNI plugin fails to initialize, causing the `NetworkPluginNotReady` error **during cluster creation** before IRSA can be configured.

The correct sequence is:
1. **Create cluster** with CNI policy on node role (CNI works via node role)
2. **Post-create**: Deploy VPC CNI IRSA role and install IRSA-enabled VPC CNI add-on (CNI switches to IRSA)
3. **Optional cleanup**: Remove CNI policy from node role (least-privilege; CNI now uses IRSA exclusively)

Step 3 is a **future optional cleanup**, not part of this change. Leaving the policy on the node role is not a security issue — the VPC CNI will prefer IRSA when available, and having the node role policy as a fallback is acceptable.

**Migration note:** For existing clusters where the node role already has `AmazonEKS_CNI_Policy`:
1. Deploy VPC CNI IRSA role
2. Install/update VPC CNI add-on to use IRSA role ARN  
3. Verify CNI functionality (pods scheduling, nodes NetworkReady)
4. (Optional) Update `iam-node-role-dev` stack to remove the policy

---

### D5 — VPC CNI IRSA role trust policy

**Decision:** `iam/vpc-cni-role.yaml` trust policy:

```yaml
AssumeRolePolicyDocument:
  Version: "2012-10-17"
  Statement:
    - Effect: Allow
      Principal:
        Federated: !Sub "arn:aws:iam::${AWS::AccountId}:oidc-provider/${OIDCIssuerHost}"
      Action: sts:AssumeRoleWithWebIdentity
      Condition:
        StringEquals:
          !Sub "${OIDCIssuerHost}:sub": "system:serviceaccount:kube-system:aws-node"
          !Sub "${OIDCIssuerHost}:aud": "sts.amazonaws.com"
```

The `OIDCIssuerHost` parameter is the OIDC issuer URL **without** the `https://` prefix (e.g., `oidc.eks.us-east-1.amazonaws.com/id/ABCD1234`).

**Rationale:** This trust policy allows only the `aws-node` service account in the `kube-system` namespace to assume the role. The audience is locked to STS to prevent token reuse outside of AWS API calls.

---

### D6 — Makefile target additions

**Decision:** Three new Makefile targets:

1. **`deploy-vpc-cni-role-dev`**: Deploys `iam-vpc-cni-role-dev` stack
   - Requires OIDC issuer URL as input (fetch from Parameter Store or cluster describe)
   - Creates change set (review required)

2. **`install-vpc-cni-addon-thor`**: Installs VPC CNI add-on on existing cluster
   - Requires VPC CNI IRSA role to exist
   - Runs `eksctl create addon` with pinned version and IRSA role ARN

3. **Update `post-create-thor`**: After OIDC association and Parameter Store write, call `deploy-vpc-cni-role-dev` then `install-vpc-cni-addon-thor`

**Rationale:** Breaking the VPC CNI setup into separate targets allows operators to debug individual steps if cluster creation fails. The `post-create-thor` target orchestrates all post-creation steps in order.

---

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| OIDC provider creation fails during cluster creation | eksctl reports error immediately; cluster creation rolls back; operator re-runs after investigating |
| VPC CNI add-on installation fails due to missing IRSA role | Makefile validates that `iam-vpc-cni-role-dev` stack exists before running `install-vpc-cni-addon-thor`; fail fast with clear error |
| Removing `AmazonEKS_CNI_Policy` from node role breaks CNI if IRSA is not working | Only remove policy after verifying VPC CNI add-on is running with IRSA and pods are successfully scheduling |
| Pinned VPC CNI version becomes outdated | Add a reminder in the runbook to check for newer VPC CNI versions quarterly; version upgrades are a separate Makefile target |

## Migration Plan

### For new clusters (this change enables):

1. Deploy `vpc-dev`, `iam-node-role-dev` (existing steps)
2. Run `make create-cluster-thor` — cluster created with OIDC provider
3. Run `make post-create-thor` — OIDC issuer stored, VPC CNI IRSA role deployed, VPC CNI add-on installed
4. Verify nodes are Ready: `kubectl get nodes`
5. Verify VPC CNI pods running: `kubectl get pods -n kube-system -l k8s-app=aws-node`

### For existing clusters with CNI error:

1. Verify cluster OIDC provider exists: `eksctl utils associate-iam-oidc-provider --cluster thor --approve` (idempotent)
2. Deploy VPC CNI IRSA role: `make deploy-vpc-cni-role-dev`
3. Install VPC CNI add-on: `make install-vpc-cni-addon-thor`
4. Wait 2-3 minutes for `aws-node` DaemonSet to roll out
5. Verify nodes report NetworkReady: `kubectl get nodes -o wide`
6. (Optional) Update node role stack to remove `AmazonEKS_CNI_Policy`

Rollback: If VPC CNI add-on installation fails, delete the add-on (`eksctl delete addon --cluster thor --name vpc-cni`) and re-run installation with corrected parameters.
